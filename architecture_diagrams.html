2 · Topologie Multi-Agents (Supervisor Pattern)

L'orchestrateur route dynamiquement vers chaque agent spécialisé. La boucle reviewer → generator est bornée par un budget de révisions.

flowchart TD START([Entrée : objectif + documents]) --> ORC ORC{{"Orchestrateur\n(Superviseur)"}} ORC -->|corpus non résumé| AN["Analyste\nDocumentaire"] ORC -->|exigences manquantes| EX["Extracteur\nd'Exigences"] ORC -->|plan vide| AR["Architecte\nde Tests"] ORC -->|cas de test manquants| GN["Générateur\nde Cas de Test"] ORC -->|traçabilité non validée| TR["Agent de\nTraçabilité"] ORC -->|review en attente| RV["Réviseur\nQualité"] RV -->|critical / major findings| ORC ORC -->|approuvé| PL["Planificateur\n(Gantt / Ressources)"] PL --> ORC ORC -->|planifié| END([Plan Final Complet]) AN --> ORC EX --> ORC AR --> ORC GN --> ORC TR --> ORC style ORC fill:#1a3a5c,color:#fff,stroke:#0d2438 style END fill:#27ae60,color:#fff style START fill:#2980b9,color:#fff style RV fill:#c0392b,color:#fff
Fast paths procéduraux : L'orchestrateur ne consume pas un appel LLM pour les décisions évidentes (pas de requirements → extractor). Le LLM est uniquement utilisé pour les branches ambiguës (faut-il relancer le generator ou passer au planner ?).

3 · Architecture Mémoire (4 Niveaux)

Chaque niveau est une interface Protocol — swappable sans toucher au code des agents.

flowchart TD subgraph T1["Tier 1 — Working Memory"] WM["Scratchpad volatile (TTL)\n• Résumé corpus en cours\n• Draft plan courant\n• Sorties d'outils récentes"] end subgraph T2["Tier 2 — Mémoire Épisodique"] EP["Journal de session ordonné\n• Chaque décision d'agent\n• Turns utilisateur\n• Findings réviseur\n• Erreurs"] end subgraph T3["Tier 3 — Mémoire Sémantique"] VS["Vector Store (namespaces)\nchunks:general\nchunks:{project_id}\nrequirements:{project_id}\nlessons"] end subgraph T4["Tier 4 — Graphe Cross-Document"] GR["NetworkX DiGraph (→ Neo4j)\nNœuds : Doc·Section·Chunk·Req·TC\nArêtes typées : derives_from\ncovers · refines · contradicts"] end T1 -.->|perd à la mort du process| T2 T2 -.->|recherche textuelle| T3 T3 -.->|ids → traversal| T4 style T1 fill:#fef9e7 style T2 fill:#eafaf1 style T3 fill:#eaf4fb style T4 fill:#fdf2f8
In-memory par défaut. Tier 3 → Qdrant / Chroma / pgvector · Tier 4 → Neo4j / Memgraph — les interfaces existent, les adapters sont la prochaine étape.

4 · Pipeline d'Ingestion Streaming

Conçu pour ne jamais charger un document entier en mémoire.

flowchart TD F["Fichier source\n(PDF 10 000 pages)"] F --> LD["Loader spécifique\n(pypdf / python-docx / openpyxl)"] LD -->|"iterator lazy\nO(1 page) RAM"| BL subgraph BL["Stream de RawBlocks"] B1["heading (level 1-10)"] B2["prose"] B3["table (atomique)"] B4["list_item"] B5["code / formula (atomique)"] end BL --> CH subgraph CH["Chunker Hiérarchique — 2 passes"] P1["Pass 1 : Arbre de sections\n(numéros, titres, hiérarchie)"] P2["Pass 2 : Packing token-bounded\n+ overlap phrase-aware\n+ blocs atomiques intacts"] P1 --> P2 end CH --> S["Sections\n(offsets char + pages)"] CH --> C["Chunks\n(token-count + back-pointer)"] C -->|"LLM fast-tier\nconcurrence ×16"| E["Exigences\n(déduplication cosinus)"] E --> MEM[("Mémoire\n+ Graphe")] C --> MEM style F fill:#fdebd0 style MEM fill:#1a3a5c,color:#fff
Clé de scalabilité : chaque loader est un générateur Python — le chunker ne voit jamais qu'un block à la fois. L'extraction LLM est parallélisée avec un sémaphore borné (×16 concurrent).

6 · Mode Autonome — LangGraph State Machine

Fire-and-forget : l'utilisateur ingère des docs, l'IA génère le plan complet.

stateDiagram-v2 [*] --> Orchestrateur Orchestrateur --> Analyste : corpus_summary absent Orchestrateur --> Extracteur : requirements vides Orchestrateur --> Architecte : plan absent Orchestrateur --> Générateur : test_cases vides Orchestrateur --> Traçabilité : trace_report absent Orchestrateur --> Réviseur : review_report absent Orchestrateur --> Planificateur : review ok Orchestrateur --> [*] : finished=true Analyste --> Orchestrateur Extracteur --> Orchestrateur Architecte --> Orchestrateur Générateur --> Orchestrateur Traçabilité --> Orchestrateur Planificateur --> Orchestrateur Réviseur --> Orchestrateur : findings critical/major\n+ budget révisions restant Réviseur --> Orchestrateur : approuvé → route planner
Le budget de révisions (max_revision_rounds) garantit la terminaison. Chaque état est sérialisable — LangGraph peut checkpointer et reprendre après une interruption.

7 · Mode Interactif — Copilot Chatbot

Chaque turn = un passage dans le graphe. Contexte grounded via retrieval hybride.

CP->>MEM: retrieve(query, project_id) Note over MEM: Semantic search (chunks + reqs)\n+ Graph neighbours MEM-->>CP: chunks + requirements\n+ graph context CP->>LLM: [system_prompt + grounding + history + user_msg] LLM-->>CP: CopilotReply(message, citations, proposed_action) CP->>MEM: log episodic (user turn + assistant turn) CP-->>U: réponse + ["spec_v3.pdf p.41", "ISO 4413 §5.2"] U->>CP: "Ajoute un test pour REQ-4.2.1" CP-->>U: proposition + needs_confirmation=true U->>CP: "Confirme" CP->>MEM: register_test_case(...) CP-->>U: "Test ajouté — tc_new9x"
Le retrieval est hybride : recherche vectorielle (similarité sémantique) + traversal graphe (liens structurels). Les mutations au plan nécessitent une confirmation explicite.

8 · Zero Vendor Lock-in — LLM Gateway Agnostique

Les agents ne connaissent que le Protocol. LiteLLM résout le fournisseur à l'exécution.

Même design pour les stores : SemanticStore Protocol → InMemory (ref) → Qdrant / Chroma / pgvector. CrossDocumentGraph → NetworkX → Neo4j.
